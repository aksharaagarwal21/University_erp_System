ABSTRACT  
   
The Smart University Management System is a database-oriented project designed to manage 
essential academic activities of a university in a simple and efficient manner. Universities 
handle large volumes of data related to students, faculty, courses, attendance, and 
examinations. Managing this data manually can lead to redundancy, inconsistency, and errors.  
   
This project focuses on designing a simplified relational database using MySQL that supports 
core university operations such as student registration, course enrollment, attendance 
tracking, examination management, and result processing. The system uses Entity 
Relationship (ER) modeling to represent real-world entities and their relationships and applies 
normalization techniques to ensure data integrity and reduce redundancy.  
   
The proposed system demonstrates the practical application of DBMS concepts such as 
relational schema design, SQL queries, constraints, and normalization, while keeping the 
implementation simple and easy to understand.

1.1 Introduction  
A University ERP (Enterprise Resource Planning) System is a centralized database system 
designed to manage all academic and administrative activities of a university in an efficient 
and organized manner.  
The system integrates multiple modules such as student management, faculty management, 
course management, examination handling, and result processing into a single database 
platform.  
In traditional systems, most university records are maintained manually or in isolated software, 
which leads to data redundancy, inconsistency, and difficulty in retrieval. The proposed 
University ERP database system eliminates these issues by providing a structured relational 
database that ensures accuracy, reliability, and quick access to information.  
  
1.2 Motivation  
The motivation behind developing the University ERP System is:  
• To reduce manual paperwork and human errors.  
• To centralize academic data in one secure database.  
• To enable fast data retrieval and reporting.  
• To maintain data integrity and consistency.  
• To support decision-making using organized information.  
Universities handle thousands of students and faculty records every year, and without an 
automated database system, management becomes inefficient. Hence, a DBMS-based ERP 
system is essential.  
  
 1.3 Scope  
 
The scope of the University ERP System includes:  
• Student registration and profile management  
8  
  
• Faculty information management  
• Department and course handling  
• Subject enrollment  
• Timetable scheduling  
• Examination and result processing  
• Report generation  
The system is limited to academic management and does not include financial or hostel 
modules in this mini project.  
 
1.4 Problem Statement  
Universities often struggle with data duplication, mismanagement of records, delayed result 
processing, and lack of centralized access. Manual or semi-automated systems cannot 
efficiently handle large volumes of data and relationships among entities such as students, 
courses, and faculty.  
Therefore, there is a need for a relational database system that can:  
• Store large amounts of structured data  
• Maintain relationships among entities  
• Provide secure and fast access  
• Generate meaningful academic reports  
The University ERP System is proposed to solve these problems using DBMS concepts such 
as ER modeling, normalization, constraints, and SQL operations.

2.1 Relational Schema for University ERP System  
  
   
Figure 2.1: Relational Schema for University ERP System  
  
Figure 2.1 depicts the relational schema of the University ERP System.  
The relational schema shows how the ER diagram is converted into structured database tables 
using Primary Keys (PK) and Foreign Keys (FK) to maintain relationships and data integrity.  
The Department table acts as a parent table and is referenced by Student, Faculty, and Course 
tables, indicating that one department can have many students, faculty members, and courses. 
The Subject table is linked to the Course table to represent subjects offered under each course.  
The Enrolment table works as a bridge table between Student and Subject, resolving the many-
to-many relationship and storing enrolment details. The Timetable table manages class 
scheduling by connecting faculty and subjects. The Exam table stores examination details, 
while the Result table records student performance by linking students with their exams.  
13  
  
Overall, this schema ensures organized data storage, reduced redundancy, and proper 
referential integrity, enabling efficient academic data management in the University ERP 
System.

5.1 Introduction to Transactions  
  
A transaction in a database is a sequence of one or more SQL operations that are executed as a 
single logical unit of work. Transactions ensure that the database moves from one consistent state to 
another, even in the presence of failures or concurrent access. 
In the University ERP System, transactions are used to ensure that operations such as student 
enrollment, result entry, and timetable updates are completed fully or not at all, preventing partial 
updates that could corrupt academic records. 
 
5.1.1 Properties of Transactions (ACID) 
Property 
Description 
Atomicity 
All operations in a transaction are completed successfully, or none are 
applied. E.g., enrolling a student and updating result must both 
succeed or both fail. 
Consistency 
The database must remain in a valid state before and after a 
transaction. Constraints like foreign keys and CHECK constraints 
must always be satisfied. 
Isolation 
Concurrent transactions execute independently without interfering 
with each other. One transaction's intermediate state is invisible to 
others. 
Durability 
Once a transaction is committed, the changes are permanently saved 
even if the system crashes immediately after.

5.2 Transaction Control Language (TCL) 
Transaction Control Language (TCL) commands are used to manage and control transactions in 
MySQL. The three main TCL commands used in this project are: 
 
5.2.1 SAVEPOINT 
A SAVEPOINT creates a marker within a transaction to which you can later roll back, without affecting 
the entire transaction. Syntax: 
SAVEPOINT savepoint_name; 
 
5.2.2 COMMIT 
COMMIT permanently saves all changes made during the current transaction. Once committed, 
changes cannot be undone with ROLLBACK. Syntax: 
COMMIT; 
 
5.2.3 ROLLBACK 
ROLLBACK undoes all changes made during the current transaction (or to a SAVEPOINT) and 
restores the database to its previous consistent state. Syntax: 
ROLLBACK; 
ROLLBACK TO savepoint_name;

5.3.1 Transaction 1: Enroll a New Student and Assign to a Subject 
This transaction inserts a new student record and enrolls the student into a subject. If any step fails, the

-- Step 3: Enroll student in a subject

  FROM Student s JOIN Enrollment e 
  ON s.StudentID = e.StudentID 
  WHERE s.StudentID = 104; 
 
-- Step 5: Commit the transaction

5.3.2 Transaction 2: Insert Exam Result with Rollback on Error 
This transaction inserts a result for a student. If the marks violate the CHECK constraint (marks > 100), 
the transaction is rolled back to the savepoint, preventing invalid data entry. 
 
-- Transaction 2: Insert Exam Result with Rollback 
START TRANSACTION; 
 
-- Step 1: Insert valid result for Sneha 
INSERT INTO Exam VALUES (704, '2026-04-10', 100, 401); 
 
SAVEPOINT after_exam_insert; 
 
-- Step 2: Insert valid result 
INSERT INTO Result VALUES (807, 78, 'B', 7.8, 104, 704); 
 
SAVEPOINT after_valid_result; 
 
-- Step 3: Attempt invalid result (marks = 115, violates CHECK constraint) 
-- This will fail: INSERT INTO Result VALUES (808, 115, 'A+', 9.5, 104, 703); 
 
-- Step 4: Rollback to savepoint before the bad insert 
ROLLBACK TO after_valid_result; 
 
-- Step 5: Verify only valid result exists 
SELECT * FROM Result WHERE StudentID = 104; 
 
-- Step 6: Commit the valid changes 
COMMIT;

5.3.3 Transaction 3: Update Enrollment Status with Savepoints 
This transaction updates the enrollment status of multiple students. If an update on one student fails, 
only that change is rolled back while the rest are committed.

-- Step 1: Deactivate enrollment for student 101 in subject 401 
UPDATE Enrollment SET Status = 'Completed' 
  WHERE StudentID = 101 AND SubjectID = 401; 
 
SAVEPOINT after_first_update;

UPDATE Enrollment SET Status = 'Completed' 
  WHERE StudentID = 102 AND SubjectID = 402;

5.4 Concurrency Control 
Concurrency control ensures that multiple transactions can execute simultaneously without causing data 
inconsistency or integrity violations. In the University ERP System, concurrency control is crucial when 
multiple users (e.g., faculty entering results, students enrolling) access the database at the same time. 
 
62  
  
5.4.1 Concurrency Control Algorithms 
MySQL implements concurrency control primarily through locking mechanisms. The main approaches 
are: 
 
Algorithm 
Description 
Two-Phase Locking 
(2PL) 
Transactions acquire all locks before releasing any. Growing phase: 
locks acquired. Shrinking phase: locks released. Guarantees 
serializability. 
Timestamp Ordering 
Each transaction is assigned a timestamp. Operations are ordered 
by these timestamps to ensure consistency. 
MVCC (Multi-Version 
Concurrency Control) 
MySQL InnoDB uses MVCC. Readers do not block writers; each 
transaction sees a snapshot of data consistent with its start time.

5.4.2 Locking Commands in MySQL 
 
a. Row-Level Locking – SELECT ... FOR UPDATE 
Locks specific rows for the duration of the transaction. Other transactions are blocked from modifying 
the locked rows until the current transaction commits or rolls back. 
-- Lock a student's result row for exclusive update 
START TRANSACTION; 
SELECT * FROM Result 
  WHERE StudentID = 101 FOR UPDATE; 
-- Perform updates on the locked row 
UPDATE Result SET Marks = 90, Grade = 'A+' 
  WHERE StudentID = 101 AND ExamID = 701; 
COMMIT; 
 
b. Table-Level Locking – LOCK TABLES 
Locks an entire table to prevent other sessions from reading or writing during critical operations such 
as bulk updates. 
-- Acquire a write lock on the Enrollment table 
63  
  
LOCK TABLES Enrollment WRITE; 
 
-- Perform bulk update 
UPDATE Enrollment SET Status = 'Archived' 
  WHERE AcademicYear = '2024'; 
 
-- Release the lock 
UNLOCK TABLES; 
 
Lock Modes Reference: 
Lock Mode 
Description 
ROW SHARE 
Allows concurrent read access; prevents exclusive table locks. 
ROW EXCLUSIVE 
Used for DML operations; prevents other sessions from locking in 
share mode. 
SHARE 
Allows queries but blocks updates and deletes from other sessions. 
SHARE ROW

Table 5.9: MySQL Lock Modes 
 
c. COMMIT – Release All Locks 
COMMIT;  -- Commits all changes and releases all row/table locks. 
 
d. ROLLBACK – Undo Changes and Release Locks 
ROLLBACK;  -- Undoes all changes since the last COMMIT and releases all locks. 
 
5.4.3 Concurrency Control Example for University ERP System 
Scenario: Two faculty members (Session A and Session B) are simultaneously trying to update the result 
of Student 101 for Exam 701. Locking ensures that only one session modifies the record at a time.

Session B (Faculty 2)

SELECT * FROM Result

-- Session B sees updated value (Marks = 92) and can now update
